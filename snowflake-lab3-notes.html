Lab 3: Caching and Query Performance

Objective: Learn how Snowflake employs caching and how to use the Query Profile to analyze query performance and caching efficiency.

Scenario: You're a data analyst at Snowbear Air learning to use the Query Profile to analyze queries you think could be running faster by better leveraging Snowflake's caching.

üìö Learning Objectives

  • How to access and navigate the Query Profile
  • Understand the differences between metadata cache, query result cache, and data cache
  • How to determine when and why the query result cache is being used or not
  • How to determine if partition pruning is efficient, and if not, how to improve it
  • How to determine if spillage is taking place
  • How to use EXPLAIN to determine how to improve your queries

1. Accessing and Navigating the Query Profile

1.1 Create Folder and Worksheet

  1. Create a new folder called Caching and Query Performance
  2. Create a new worksheet inside the folder called Working with Cache and the Query Profile

1.2 Set Context and Clear Cache

Set the worksheet context and clear any cached data from previous sessions:

USE ROLE TRAINING_ROLE;
USE WAREHOUSE WOODCHUCK_WH;
CREATE DATABASE IF NOT EXISTS WOODCHUCK_DB;
USE SCHEMA WOODCHUCK_DB.PUBLIC;

-- This will clear the data from previous sessions that is cached in the virtual warehouse
ALTER WAREHOUSE WOODCHUCK_WH SUSPEND;
ALTER WAREHOUSE WOODCHUCK_WH RESUME;

1.3 Create Sample Data

Create a table with customer data and suspend the warehouse to ensure subsequent queries don't use cached data:

CREATE TABLE customer AS
SELECT
    c_custkey,
    c_firstname,
    c_lastname
FROM
    SNOWBEARAIR_DB.PROMO_CATALOG_SALES.customer;

ALTER WAREHOUSE WOODCHUCK_WH SUSPEND;

1.4 Execute Query and Access Query Profile

Run a simple query to generate results:

SELECT DISTINCT *
FROM customer c;

To access the Query Profile:

  1. Close the "Result limit exceeded" panel (click the X)
  2. Click the ellipsis (‚ãØ) in the Query Details panel
  3. Select View Query Profile
  4. The Query Profile opens in a new tab

üí° Understanding the Query Profile

The Query Profile provides two main views:

  • Profile Overview: High-level statistics about the query execution
  • Operator Tree: Detailed breakdown of each operation in the query

2. Understanding Snowflake Caching

2.1 Types of Caching

Snowflake employs three types of caching:

  • Query Result Cache: Stores complete query results for 24 hours (requires identical query)
  • Metadata Cache: Stores table metadata like min/max values, row counts, and structure
  • Data Cache: Stores actual data in the warehouse's local SSD storage

2.2 Query Result Cache

The query result cache returns results instantly if:

  • The exact same query was run within the last 24 hours
  • The underlying data hasn't changed
  • No changes to table structure or permissions

Test query result caching:

-- Run this query twice and compare Query Profile results
SELECT DISTINCT *
FROM customer c;

On the second run, check the Query Profile - you should see "Query result reuse" indicating the result cache was used.

2.3 Metadata Cache

Metadata cache helps Snowflake quickly determine which micro-partitions to scan without reading actual data.

Example query that benefits from metadata cache:

-- Aggregation queries often use metadata cache
SELECT COUNT(*) FROM customer;

2.4 Data Cache

Data cache stores actual data in the warehouse's SSD. When you suspend and resume a warehouse, the data cache is cleared.

Test data caching:

-- Clear cache
ALTER WAREHOUSE WOODCHUCK_WH SUSPEND;
ALTER WAREHOUSE WOODCHUCK_WH RESUME;

-- Run query (will read from remote storage)
SELECT * FROM customer LIMIT 1000;

-- Run same query again (will use data cache)
SELECT * FROM customer LIMIT 1000;

Compare the Query Profiles - the second query should show faster execution with "Percentage scanned from cache" closer to 100%.

3. Partition Pruning

Partition pruning is when Snowflake skips scanning unnecessary micro-partitions based on filter predicates.

3.1 Analyzing Partition Pruning Efficiency

In the Query Profile, look for:

  • Partitions scanned: Number of micro-partitions actually read
  • Partitions total: Total number of micro-partitions in the table

Efficient partition pruning means scanning far fewer partitions than the total.

‚ö†Ô∏è Pro Tip

Use clustered columns in your WHERE clause to maximize partition pruning efficiency. Filters on clustered columns allow Snowflake to skip more partitions.

4. Detecting Spillage

Spillage occurs when query operations exceed available memory and must write intermediate results to disk (local or remote storage).

4.1 Types of Spillage

  • Local Spillage: Writing to local SSD (slower than memory)
  • Remote Spillage: Writing to remote storage (much slower)

4.2 Finding Spillage in Query Profile

In the Query Profile Operator Tree, look for:

  • Bytes spilled to local storage
  • Bytes spilled to remote storage

If you see significant spillage, consider:

  • Increasing warehouse size (more memory per node)
  • Optimizing query to reduce memory requirements
  • Breaking complex queries into smaller steps

5. Using EXPLAIN Plans

The EXPLAIN command shows the execution plan without actually running the query.

5.1 Generate an EXPLAIN Plan

EXPLAIN
SELECT DISTINCT *
FROM customer c;

The EXPLAIN plan shows:

  • Query compilation steps
  • Estimated costs
  • Operations that will be performed
  • Partition pruning estimates

6. Key Takeaways

  • Snowflake has three types of caching: query result cache, metadata cache, and data cache
  • The Query Profile is essential for understanding query performance and caching behavior
  • Query result cache provides instant results for identical queries within 24 hours
  • Metadata cache helps with aggregation queries and partition pruning
  • Data cache stores actual data in warehouse SSD for faster subsequent access
  • Partition pruning efficiency depends on using appropriate filter predicates
  • Spillage (especially remote spillage) indicates memory pressure - consider larger warehouse
  • EXPLAIN plans help predict query behavior before execution
