Time to Code

We have read about architectures, communication models, and the tradeoffs between centralized and decentralized systems. Now we build them.

The labs for this week are hands-on Google Colab notebooks that run real code against real cloud infrastructure. You will write Python, send actual messages through AWS, and observe what happens when distributed systems behave -- and when they do not.

There are two tracks. You will do them in order.

Track 1: Queues and Pub/Sub

Start here. This notebook walks you through message queues (Amazon SQS), publish-subscribe messaging (Amazon SNS), and what happens when you try to count things in a distributed system.

You will go from sending your first message to detecting duplicates, traffic spikes, and timestamp fraud -- and then see what happens when three independent auditors try to agree on a number.

Sections covered:

  1. I Can Talk Using Queues -- SQS basics, FIFO ordering, visibility timeout
  2. Pub/Sub Concepts -- SNS topics, fan-out, decoupling
  3. I Am a Host -- defining and publishing ViewEvents in the 2032 system
  4. Guarantees and Their Limits -- idempotent processing, FIFO vs. standard
  5. I Am Not Alone -- multi-host traffic, filtering, observation
  6. Something Is Wrong -- detecting duplicates, spikes, late events
  7. Multiple Auditors -- comparing independent counts, majority voting

Track 2: Peer-to-Peer

After you finish the pub/sub lab, you will have seen the limits of centralized messaging firsthand. The auditors could not talk to each other. Nobody checked for consensus. There was no way to discover who else was in the network.

The peer-to-peer track changes that. You will learn -- and implement -- the real algorithms that power systems like BitTorrent, Cassandra, and Ethereum: gossip protocols for discovery, heartbeats for liveness, choking for fairness, Chord for distributed lookup, and reputation scoring for trust.

By the end, each of you will have built a working P2P node. Every node in the class will be part of a single live network. Your nodes will discover each other, communicate directly, publish events, audit each other, and agree on view counts -- with no central server, no broker, no authority.

Two notebooks for this track:

  • P2P Algorithms Lab -- learn and implement each algorithm in isolation (gossip, heartbeat, choking, Chord, reputation)
  • P2P Protocol Lab -- wire the algorithms together into a working node that joins a real network
