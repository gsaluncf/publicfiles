Lab 7: Introduction to Monitoring Usage and Billing

Objective: Learn to use the SNOWFLAKE database and ACCOUNT_USAGE schema to monitor object usage, track billing metrics, and optimize credit consumption.

Scenario: As a Snowflake administrator, you need to understand how credits are being consumed and identify opportunities to optimize costs.

üìö Learning Objectives

  • Monitor usage and billing with the ACCOUNT_USAGE schema
  • Determine which warehouses do not have resource monitors activated
  • Identify the most expensive queries from the last 30 days
  • Find the top 10 queries with the most spillage to remote storage
  • Understand billing metrics and credit consumption patterns

üí° About the SNOWFLAKE Database

Snowflake provides a system-defined, read-only shared database named SNOWFLAKE that contains metadata and historical usage data about objects in your organization and account.

ACCOUNT_USAGE schema contains secure views for:

  • Object usage history
  • Grants to roles and users
  • Data loading history
  • Metering and billing history
  • Storage history
  • Query history

2. Monitoring Credit Usage

2.1 Examine Credit Usage by Warehouse (All Time)

View total credit consumption for each warehouse over the past year:

-- Credits used (all time = past year)
SELECT 
    WAREHOUSE_NAME,
    SUM(CREDITS_USED_COMPUTE) AS CREDITS_USED_COMPUTE_SUM
FROM 
    ACCOUNT_USAGE.WAREHOUSE_METERING_HISTORY
GROUP BY 1
ORDER BY 2 DESC;

2.2 Credits Used in Past N Days

Analyze credit consumption for a specific time period:

-- Credits used (past 7 days)
SELECT 
    WAREHOUSE_NAME,
    SUM(CREDITS_USED_COMPUTE) AS CREDITS_USED_COMPUTE_SUM
FROM 
    ACCOUNT_USAGE.WAREHOUSE_METERING_HISTORY
WHERE 
    START_TIME >= DATEADD(DAY, -7, CURRENT_TIMESTAMP())
GROUP BY 1
ORDER BY 2 DESC;

üí° Pro Tip: Column Position

Notice GROUP BY 1 and ORDER BY 2 - you can group or order by column position instead of repeating long column names!

2.3 Analyzing Credit Consumption

Use these queries to:

  • Identify warehouses consuming more credits than expected
  • Spot trends in credit usage over time
  • Investigate why specific warehouses have high consumption

If a warehouse is consuming too many credits, consider:

  • Modifying the auto-suspend policy
  • Adjusting the scaling policy
  • Reviewing data loading efficiency
  • Analyzing query performance and optimization
  • Adding a resource monitor

3. Resource Monitors

3.1 Find Warehouses Without Resource Monitors

Resource monitors limit credit consumption during a specific time interval or date range:

SHOW WAREHOUSES;

SELECT 
    "name" AS WAREHOUSE_NAME,
    "size" AS WAREHOUSE_SIZE
FROM 
    TABLE(RESULT_SCAN(LAST_QUERY_ID()))
WHERE 
    "resource_monitor" = 'null';

This identifies warehouses that don't have resource monitors, which could lead to unexpected credit consumption.

üìñ About RESULT_SCAN

The RESULT_SCAN() function allows you to query the results of a previous statement (like SHOW WAREHOUSES) as if it were a table.

4. Billing Metrics

4.1 Most Expensive Queries (Last 30 Days)

Identify queries that consumed the most credits:

SELECT
    query_id,
    query_text,
    user_name,
    warehouse_name,
    warehouse_size,
    (execution_time / 1000) AS execution_time_seconds,
    credits_used_cloud_services
FROM
    ACCOUNT_USAGE.QUERY_HISTORY
WHERE
    start_time >= DATEADD(DAY, -30, CURRENT_TIMESTAMP())
    AND execution_status = 'SUCCESS'
    AND warehouse_size IS NOT NULL
ORDER BY
    credits_used_cloud_services DESC
LIMIT 20;

This helps identify:

  • Long-running queries that consume many credits
  • Queries that might benefit from optimization
  • Users or warehouses generating high costs

4.2 Queries with Most Spillage to Remote Storage

Spillage to remote storage is expensive and slow. Find the top offenders:

SELECT
    query_id,
    user_name,
    warehouse_name,
    warehouse_size,
    bytes_spilled_to_remote_storage,
    bytes_spilled_to_local_storage,
    (execution_time / 1000) AS execution_time_seconds,
    query_text
FROM
    ACCOUNT_USAGE.QUERY_HISTORY
WHERE
    start_time >= DATEADD(DAY, -30, CURRENT_TIMESTAMP())
    AND bytes_spilled_to_remote_storage > 0
ORDER BY
    bytes_spilled_to_remote_storage DESC
LIMIT 10;

For queries with significant remote spillage, consider:

  • Increasing warehouse size (provides more memory)
  • Rewriting the query to be more memory-efficient
  • Breaking complex operations into multiple steps
  • Using materialized views or intermediate tables

4.3 Storage Costs

Monitor your storage consumption:

SELECT
    usage_date,
    storage_bytes / (1024*1024*1024) AS storage_gb,
    stage_bytes / (1024*1024*1024) AS stage_gb,
    failsafe_bytes / (1024*1024*1024) AS failsafe_gb
FROM
    ACCOUNT_USAGE.STORAGE_USAGE
ORDER BY
    usage_date DESC
LIMIT 30;

4.4 Daily Credit Consumption Trend

Track credit consumption over time to identify patterns:

SELECT
    DATE(start_time) AS usage_date,
    warehouse_name,
    SUM(credits_used_compute) AS total_credits
FROM
    ACCOUNT_USAGE.WAREHOUSE_METERING_HISTORY
WHERE
    start_time >= DATEADD(DAY, -30, CURRENT_TIMESTAMP())
GROUP BY 1, 2
ORDER BY 1 DESC, 3 DESC;

5. Advanced Analysis

5.1 User Activity Analysis

-- Top users by query count
SELECT
    user_name,
    COUNT(*) AS query_count,
    SUM(execution_time) / 1000 AS total_execution_seconds
FROM
    ACCOUNT_USAGE.QUERY_HISTORY
WHERE
    start_time >= DATEADD(DAY, -7, CURRENT_TIMESTAMP())
GROUP BY 1
ORDER BY 2 DESC
LIMIT 10;

5.2 Failed Queries Analysis

-- Recent failed queries
SELECT
    query_id,
    user_name,
    error_code,
    error_message,
    query_text
FROM
    ACCOUNT_USAGE.QUERY_HISTORY
WHERE
    start_time >= DATEADD(DAY, -7, CURRENT_TIMESTAMP())
    AND execution_status = 'FAIL'
ORDER BY
    start_time DESC
LIMIT 20;

6. Key Takeaways

  • The SNOWFLAKE database contains metadata and usage information for your entire account
  • ACCOUNT_USAGE schema provides secure views for monitoring usage and billing
  • Use WAREHOUSE_METERING_HISTORY to track credit consumption by warehouse
  • Resource monitors help control credit consumption and prevent unexpected costs
  • QUERY_HISTORY view provides detailed information about all executed queries
  • Monitor spillage to remote storage - it's slow and expensive
  • Use GROUP BY and ORDER BY with column positions (1, 2, 3) for cleaner queries
  • Regular monitoring helps identify optimization opportunities and cost savings
  • DATEADD() function is useful for filtering data by time periods
  • RESULT_SCAN() allows querying results from SHOW commands
