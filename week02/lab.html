Lab: Deploy a Static Website to AWS S3

This lab guides you through deploying a static website to Amazon S3 using both AWS Console UI and AWS CLI methods. You'll learn the complete workflow from bucket creation to public website deployment.

Time estimate: 45-60 minutes

This lab provides two complete methods for deploying your S3 website:

Method 1: AWS Console UI

This method uses the AWS web interface. Follow these steps carefully, referencing the official AWS tutorial.

Step 1: Create an S3 Bucket

  1. Sign in to the AWS S3 Console
  2. Click Create bucket
  3. Enter a globally unique bucket name:
    • Format: ncf-s26-yourname-site
    • Example: ncf-s26-gsalu-demo-site
    • Must be lowercase, no spaces, globally unique across all AWS
  4. Choose AWS Region: us-east-1 (recommended for simplicity)
  5. Leave all other settings as default
  6. Click Create bucket

Why bucket names must be globally unique

S3 bucket names form part of the URL (e.g., bucket-name.s3.amazonaws.com). Since URLs must be unique, bucket names must be unique across all AWS accounts worldwide.

Step 2: Enable Static Website Hosting

  1. In the S3 console, click on your newly created bucket name
  2. Click the Properties tab
  3. Scroll down to Static website hosting
  4. Click Edit
  5. Select Enable
  6. Under Hosting type, choose Host a static website
  7. Enter Index document: index.html
  8. Enter Error document: index.html (for single-page apps)
  9. Click Save changes
  10. Important: Note the Bucket website endpoint URL shown at the bottom (you'll need this later)

Website Endpoint Format

Your website endpoint will look like:

http://bucket-name.s3-website-region.amazonaws.com

Example: http://ncf-s26-gsalu-demo-site.s3-website-us-east-1.amazonaws.com

Step 3: Upload Website Files

  1. In your cloned repository, navigate to S3WorkingArea/site/
  2. You should see three files:
    • index.html ‚Äî Main page
    • styles.css ‚Äî Styling
    • script.js ‚Äî JavaScript
  3. Back in the S3 console, click the Objects tab
  4. Click Upload
  5. Click Add files and select all three files from the site/ directory
  6. Click Upload
  7. Wait for upload to complete, then click Close

Step 4: Edit Block Public Access Settings

‚ö†Ô∏è Security Warning

By default, AWS blocks all public access to S3 buckets for security. To host a public website, you must disable this protection. Only do this for buckets intended to be public websites.

  1. In your bucket, click the Permissions tab
  2. Under Block public access (bucket settings), click Edit
  3. Uncheck Block all public access
  4. Click Save changes
  5. Type confirm in the confirmation box
  6. Click Confirm

Step 5: Add Bucket Policy for Public Read Access

Disabling Block Public Access isn't enough ‚Äî you must also add a bucket policy that explicitly grants public read permissions.

  1. Still in the Permissions tab, scroll to Bucket policy
  2. Click Edit
  3. Paste the following policy (replace YOUR-BUCKET-NAME with your actual bucket name):
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "PublicReadGetObject",
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::YOUR-BUCKET-NAME/*"
    }
  ]
}
  1. Click Save changes

Understanding the Bucket Policy

  • "Principal": "*" ‚Äî Allows anyone (public access)
  • "Action": "s3:GetObject" ‚Äî Allows reading objects (not writing or deleting)
  • "Resource": ".../*" ‚Äî Applies to all objects in the bucket

Step 6: Test Your Website

  1. Go back to the Properties tab
  2. Scroll to Static website hosting
  3. Click the Bucket website endpoint URL
  4. Your website should open in a new tab!

If you see your website, congratulations! You've successfully deployed a static website to AWS S3 using the Console UI.

Method 2: AWS CLI Scripts (Recommended)

This method uses pre-built, tested automation scripts. These scripts have been verified to work and provide a faster, more reliable deployment process.

Why Use CLI Scripts?

  • Tested and verified: All scripts have been run successfully and produce working websites
  • Faster: Complete deployment in minutes instead of clicking through multiple screens
  • Repeatable: Easy to tear down and redeploy if needed
  • Automation practice: Learn infrastructure-as-code concepts
  • Error handling: Scripts check for common issues and provide helpful error messages

Step-by-Step CLI Deployment

Step 1: Verify AWS CLI Setup

Open PowerShell or Command Prompt and navigate to your repository:

cd path\to\your\repository\S3WorkingArea
.\00_verify_aws_cli.bat

This script checks:

  • AWS CLI is installed
  • Credentials are configured
  • You have S3 access permissions

Troubleshooting: If verification fails

  • "AWS CLI not found": Install AWS CLI from aws.amazon.com/cli
  • "Unable to locate credentials": Run aws configure and enter your access keys
  • "SignatureDoesNotMatch": Your credentials are invalid. Run aws configure with new access keys

Step 2: Create S3 Bucket

.\01_create_bucket.bat

The script will prompt you for:

  • Bucket name: Enter a globally unique name (e.g., ncf-s26-yourname-site)
  • AWS region: Press Enter for default (us-east-1) or enter your preferred region

What this script does:

  1. Creates the S3 bucket in your chosen region
  2. Enables static website hosting
  3. Sets index.html as the index document
  4. Saves bucket name and region to bucket_config.txt for other scripts to use

Step 3: Upload Website Files

.\02_sync_site.bat

What this script does:

  • Syncs all files from site/ directory to your S3 bucket
  • Uses aws s3 sync command with --delete flag
  • Uploads: index.html, styles.css, script.js

Key Command: aws s3 sync

aws s3 sync .\site s3://bucket-name --delete

What it does:

  • sync ‚Äî Uploads new/changed files, skips unchanged files
  • --delete ‚Äî Removes files from S3 that don't exist locally
  • Much faster than uploading all files every time

Step 4: Configure Public Access

.\03_set_bucket_policy.bat

What this script does:

  1. Disables Block Public Access settings
  2. Creates a bucket policy JSON file
  3. Applies the policy to allow public read access (s3:GetObject)

‚ö†Ô∏è Security Note

This script makes your bucket publicly readable. This is required for static websites, but be careful:

  • Only upload files you want to be public
  • Don't upload sensitive data, credentials, or private information
  • The policy only allows reading (GET), not writing or deleting

Step 5: Verify and Test Website

.\04_verify_website.bat

What this script does:

  • Displays your website endpoint URL
  • Tests HTTP connectivity
  • Lists bucket contents
  • Shows website configuration
  • Optionally opens the website in your browser

Troubleshooting Common Issues

Issue: "Bucket name already exists"

Cause: Bucket names are globally unique across all AWS accounts.

Solution: Choose a different name. Try adding your initials or a random number.

Issue: "403 Forbidden" when accessing website

Cause: Bucket policy not applied or Block Public Access still enabled.

Solution:

  1. Run .\03_set_bucket_policy.bat again
  2. Verify Block Public Access is disabled in Console
  3. Check bucket policy is applied correctly

Issue: "SignatureDoesNotMatch" error

Cause: AWS credentials are invalid or expired.

Solution: Run aws configure and enter new access keys from IAM console.

Issue: "Unable to locate credentials"

Cause: AWS CLI not configured.

Solution: Run aws configure and enter your access key ID, secret access key, region, and output format.

Issue: Cannot delete bucket

Cause: Bucket must be empty before deletion.

Solution: The 99_teardown.bat script handles this automatically. If doing manually:

aws s3 rm s3://bucket-name --recursive
aws s3api delete-bucket --bucket bucket-name
