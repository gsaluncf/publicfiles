Understanding AWS and Cloud Storage

Before we dive into S3, let's understand what AWS is, why it matters for data scientists, and why we're starting with storage rather than compute.

What is AWS (Amazon Web Services)?

AWS is Amazon's cloud computing platform — essentially, you rent computing resources (servers, storage, databases, etc.) from Amazon's massive data centers instead of buying and maintaining your own hardware.

The Cloud Computing Shift

Traditional approach (pre-2006):

  • Buy physical servers (expensive upfront costs)
  • Set up and maintain your own data center
  • Pay for peak capacity even when idle
  • Wait weeks/months for new hardware

Cloud approach (AWS, Azure, Google Cloud):

  • Rent exactly what you need, when you need it
  • Someone else handles hardware, security, maintenance
  • Pay only for what you use (pay-as-you-go)
  • Spin up resources in seconds, not weeks

Why AWS Matters for Data Scientists

Modern data science happens in the cloud because:

  • Datasets are too big for your laptop (multi-terabyte training data)
  • Processing is too slow on local machines (distributed computing needed)
  • Collaboration requires shared infrastructure (team access to data/models)
  • Production systems need reliability (24/7 uptime, auto-scaling)
  • Cost efficiency — rent a $50,000 GPU cluster for 2 hours instead of buying it

Real-world examples:

  • Training large language models (GPUs/TPUs in the cloud)
  • Processing genomic data (petabyte-scale analysis)
  • Running A/B tests on millions of users (distributed systems)
  • Serving ML models to mobile apps (auto-scaling APIs)

Why Start with Storage (S3) Instead of Compute?

We're starting with S3 (storage) rather than EC2 (servers) because:

  1. Simpler mental model: Storage is easier to understand than virtual servers
  2. Immediate results: Upload a file, get a URL — instant gratification
  3. Lower cost/risk: Forgetting to delete a file costs pennies; forgetting to stop a server costs dollars per hour
  4. Foundation for everything: Every AWS service reads/writes data somewhere — usually S3
  5. Real-world relevance: Data scientists interact with S3 daily, even if they never touch EC2

The learning path:

  • Week 2 (now): S3 — store and serve static files
  • Week 3-4: Databases — structured data storage
  • Week 5+: Compute (EC2, Lambda) — processing that data
  • Later: Distributed systems (Spark, EMR) — big data processing

S3's Role in Data Workflows

In professional data science, S3 is typically used for:

  • Data lakes: Central repository for raw data (CSV, Parquet, JSON, images)
  • Model artifacts: Trained models, checkpoints, experiment logs
  • ETL pipelines: Input/output for Spark, Airflow, Glue jobs
  • Backup and archival: Long-term storage with automatic lifecycle management
  • Static assets: Model documentation, dashboards, visualization exports

This lab focuses on the simplest use case (static website hosting), but the concepts apply directly to data engineering workflows.

What is AWS S3?

Amazon S3 (Simple Storage Service) is AWS's object storage service — a massive, globally distributed file system in the cloud that can store and retrieve any amount of data from anywhere on the web.

Official documentation: AWS S3 User Guide | S3 Product Page

Key Characteristics

  • Store any type of file: HTML, CSS, JavaScript, images, videos, data files, backups, etc.
  • Access via URLs: Every object gets a unique URL for HTTP/HTTPS access
  • Pay-as-you-go: Only pay for storage used + requests made + data transferred
  • 99.999999999% durability: That's 11 nines — your files won't get lost (S3 Storage Classes)
  • Automatic scaling: No capacity planning needed — S3 grows with your needs
  • Global availability: Choose from 30+ AWS regions worldwide (AWS Regions)

Beyond S3: Other AWS Services You'll Use

Compute

  • Amazon EC2 — virtual servers you control. Run long‑lived apps, custom environments, or GPU instances for ML. EC2 product page
  • AWS Lambda — serverless functions that run on demand. Great for event‑driven tasks and APIs without managing servers. Lambda product page
  • Amazon ECS/EKS — container orchestration (ECS) and managed Kubernetes (EKS) for microservices. Containers on AWS

Databases and Storage

  • Amazon RDS — managed relational databases (PostgreSQL, MySQL, etc.). RDS product page
  • Amazon DynamoDB — serverless NoSQL key‑value database with single‑digit ms latency. DynamoDB product page
  • Amazon Aurora — high‑performance MySQL/PostgreSQL‑compatible database. Aurora product page
  • Amazon EFS — shared POSIX file system for Linux workloads. EFS product page

Data Integration and Orchestration

  • AWS Glue — serverless ETL and data catalog for analytics. Glue product page
  • AWS Step Functions — visual workflows to coordinate Lambda/EC2/Glue jobs. Step Functions product page
  • Amazon EventBridge — event bus to connect apps and AWS services. EventBridge product page

Analytics and Big Data

  • Amazon Athena — serverless SQL on S3 (pay per query). Athena product page
  • Amazon EMR — managed Hadoop/Spark for large‑scale data processing. EMR product page
  • Amazon Redshift — cloud data warehouse for BI/analytics. Redshift product page
  • Amazon Kinesis — real‑time data streaming and processing. Kinesis product page

Machine Learning and AI

  • Amazon SageMaker — end‑to‑end platform to build, train, and deploy ML models. SageMaker product page
  • Amazon Bedrock — foundation models and generative AI APIs (serverless). Bedrock product page
  • Amazon Comprehend — NLP for text analysis (entities, sentiment). Comprehend product page

Networking, Delivery, and APIs

  • Amazon CloudFront — global CDN for low‑latency content delivery with HTTPS & custom domains. CloudFront product page
  • Amazon Route 53 — DNS, domain registration, and traffic routing. Route 53 product page
  • Amazon API Gateway — build and manage secure APIs at scale. API Gateway product page

Security, Identity, and Monitoring

  • AWS IAM — users, roles, and permissions for secure access control. IAM product page
  • AWS KMS — manage encryption keys for data at rest/in transit. KMS product page
  • AWS Secrets Manager — store and rotate app secrets safely. Secrets Manager product page
  • Amazon CloudWatch — metrics, logs, alarms, dashboards. CloudWatch product page
  • AWS CloudTrail — audit trail of API calls in your account. CloudTrail product page
  • AWS Cost Explorer & Budgets — track and control spend. Cost Explorer | Budgets

When to Choose What (Quick Cheats)

  • EC2 vs. Lambda: Long‑running, custom OS → EC2. Event‑driven, short tasks → Lambda.
  • RDS vs. DynamoDB: Joins/SQL/transactions → RDS. High scale, key‑value, serverless → DynamoDB.
  • Athena vs. Redshift: Ad‑hoc SQL directly on S3 → Athena. BI warehouse with fast repeated queries → Redshift.
  • S3 website vs. CloudFront: S3 is fine to start; add CloudFront for HTTPS, speed, and custom domains.
