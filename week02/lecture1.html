Answer: Client-Only Implementation

This simulator is implemented entirely in the browser using JavaScript. All physics calculations happen on your computer (the client). There is:

  • No server performing calculations
  • No database storing your trajectory data
  • No API calls being made
  • No persistent state ‚Äî refresh and everything resets

How COULD We Implement This with Client/Server Architecture?

Let's explore different architectural patterns for implementing the same application:

Figure: Basic client-server architecture - the client (web browser) requests resources from the server (source: Mozilla Developer Network)

Option 1: Client-Only (Current Implementation)

Architecture: Browser does everything

  • Pros: Fast, no server costs, works offline, simple
  • Cons: No data persistence, no sharing, limited by browser capabilities
  • Use case: Simple calculators, games, visualizations

Option 2: Client + API Server

Architecture: Browser sends parameters to server, server calculates trajectory, returns results

  • Pros: Can log usage, validate inputs, rate-limit requests, hide calculation logic
  • Cons: Slower (network latency), requires server infrastructure, costs money
  • Use case: When you need server-side validation or want to protect intellectual property

Example API call:

POST /api/calculate-trajectory
{
  "angle": 45,
  "speed": 50,
  "gravity": 9.8
}

Response:
{
  "points": [[0,0], [5,12.5], [10,20], ...],
  "maxHeight": 31.9,
  "range": 255.1
}

Option 3: Client + Database

Architecture: Browser calculates locally, saves trajectories to database for later retrieval

  • Pros: Persistent storage, can share trajectories, build history
  • Cons: Requires backend, authentication, data management
  • Use case: When users need to save and retrieve their work

Example workflow:

  1. User fires cannon (client calculates)
  2. User clicks "Save Trajectory"
  3. Client sends trajectory data to API: POST /api/trajectories
  4. Server stores in database (PostgreSQL, DynamoDB, etc.)
  5. Later: User retrieves saved trajectories: GET /api/trajectories

Option 4: Client + Compute Server for Physics

Architecture: Browser sends parameters, server runs complex physics simulation, returns results

  • Pros: Can handle complex simulations beyond browser capabilities, leverage GPUs/specialized hardware
  • Cons: Expensive, slower, requires significant infrastructure
  • Use case: Computational fluid dynamics, molecular dynamics, weather simulation

Real-world example: A climate modeling application where:

  • Browser provides UI for setting parameters
  • Server runs multi-hour simulation on GPU cluster
  • Results streamed back to browser for visualization

Part 2: Types of Distributed Systems

Now that we understand client-server patterns, let's zoom out and look at how systems can be organized at a higher level. There are three fundamental architectures for distributed systems:

Figure: Three fundamental architectures for distributed systems

1. Centralized (Single Node)

Structure: One central server handles all requests

  • Characteristics: Single point of control, all data flows through one node
  • Pros: Simple to build and reason about, consistent state, easy to debug
  • Cons: Single point of failure, limited scalability, bottleneck under load
  • Examples: Traditional web server, single database, your laptop

Real-world analogy: A single cashier at a store ‚Äî simple, but creates a line when busy

2. Distributed

Structure: Multiple nodes coordinated by a central authority

  • Characteristics: Central coordinator delegates work to worker nodes
  • Pros: Scalable (add more workers), fault-tolerant (workers can fail), efficient (parallel processing)
  • Cons: Coordinator is still a single point of failure, more complex than centralized
  • Examples: AWS (you interact with AWS services, they manage the infrastructure), Spark clusters, load-balanced web servers

Real-world analogy: A restaurant with one host (coordinator) and multiple servers (workers)

This is what AWS provides: You make requests to AWS APIs, and AWS distributes the work across their massive infrastructure

3. Decentralized

Structure: No central authority ‚Äî all nodes are equal peers

  • Characteristics: Peer-to-peer communication, consensus mechanisms, no single point of control
  • Pros: Highly fault-tolerant (no single point of failure), censorship-resistant, democratic
  • Cons: Complex to build, slower consensus, harder to coordinate, potential for conflicts
  • Examples: Bitcoin, BitTorrent, IPFS, some blockchain systems

Real-world analogy: A potluck dinner where everyone brings food and coordinates directly with each other

üí≠ Thought Experiment: The Two Generals Problem

Imagine this scenario:

The Setup:

Two generals are camped on opposite sides of an enemy city. They need to attack at the same time to win. If only one attacks, they'll be defeated.

The Problem:

They can only communicate by sending messengers through enemy territory. Any messenger might be captured.

  • General A sends a message: "Attack at dawn"
  • But how does General A know the message was received?
  • General B could send back: "Message received, will attack at dawn"
  • But how does General B know that message was received?
  • General A could send: "Acknowledged your acknowledgment"
  • But how does General A know that was received?
  • ...and so on, forever

The Question: Is there a protocol that guarantees both generals attack at the same time, even with unreliable communication?

Take a moment to think about this. Can you design a solution?

What Happens When You Break Things Apart

The Two Generals Problem illustrates a fundamental challenge in distributed systems:

  • Communication is unreliable: Networks fail, messages get lost, servers crash
  • Coordination is hard: How do you ensure multiple computers agree on the same state?
  • Perfect synchronization is impossible: There's no way to guarantee 100% certainty in distributed systems
  • Trade-offs are inevitable: You must choose between consistency, availability, and partition tolerance

Why this matters for data science:

  • When you use AWS, your data is distributed across multiple servers and data centers
  • Understanding these limitations helps you design better systems
  • You'll encounter concepts like "eventual consistency" and "at-least-once delivery"

Part 3: What is "The Cloud"?

Official Definition from AWS

According to Amazon Web Services:

"Cloud computing is the on-demand delivery of compute power, database, storage, applications, and other IT resources through a cloud services platform via the internet with pay-as-you-go pricing."

Source: AWS Whitepapers - What is cloud computing?

Breaking Down the Definition

Let's unpack the key concepts:

1. On-Demand Delivery

  • Traditional approach: Buy a server, wait weeks for delivery, install it, configure it
  • Cloud approach: Click a button, get a server in seconds
  • Example: Need a GPU for training a model? Spin one up in 30 seconds, use it for 2 hours, shut it down

2. Pay-As-You-Go Pricing

  • Traditional approach: Buy a $50,000 server upfront, pay for electricity 24/7 whether you use it or not
  • Cloud approach: Pay only for what you use, by the hour or by the second
  • Example: S3 storage costs ~$0.023 per GB per month. Store 10 GB = $0.23/month. Delete it = $0

3. Via the Internet

  • Traditional approach: Physical access to servers in your office or data center
  • Cloud approach: Access everything through web APIs and management consoles
  • Example: Create an S3 bucket from your laptop in Florida, access it from a server in Tokyo

The Cloud is NOT Magic ‚Äî It's Just Someone Else's Computers

Important Reality Check:

"The cloud" is a marketing term for renting computing resources from a company that owns massive data centers. When you use AWS:

  • Your data is stored on physical hard drives in Amazon's data centers
  • Your code runs on physical servers (or virtual machines on physical servers)
  • Your requests travel over physical network cables

The "magic" is in the abstraction ‚Äî you don't have to think about the physical infrastructure. You just use APIs.

The Business Case: OpEx vs. CapEx (Why CFOs Love the Cloud)

There's a critical financial reason why cloud computing exploded in popularity beyond just technical benefits: tax implications.

Capital Expenses (CapEx) ‚Äî Traditional IT

  • What it is: Buying physical assets (servers, storage, networking equipment)
  • Tax treatment: Must be depreciated over 3-7 years (in the US)
  • Example: Buy a $100,000 server ‚Üí Can only deduct ~$14,000-$33,000 per year
  • Cash flow impact: Pay $100,000 upfront, but tax benefit spreads over years
  • Accounting: Goes on the balance sheet as an asset

Operational Expenses (OpEx) ‚Äî Cloud Computing

  • What it is: Renting services (AWS, Azure, SaaS subscriptions)
  • Tax treatment: Fully deductible in the year incurred
  • Example: Spend $100,000 on AWS this year ‚Üí Deduct the full $100,000 this year
  • Cash flow impact: Pay as you go, immediate tax benefit
  • Accounting: Expense on the income statement (not an asset)

Why This Matters for Businesses

Scenario: A company needs $100,000 worth of computing resources

Option A: Buy servers (CapEx)

  • Pay $100,000 upfront
  • Deduct ~$20,000/year for 5 years
  • At 21% corporate tax rate: Save ~$4,200/year in taxes
  • Stuck with the hardware even if needs change

Option B: Use AWS (OpEx)

  • Pay $100,000 over the year (or less if you scale down)
  • Deduct the full $100,000 this year
  • At 21% corporate tax rate: Save $21,000 in taxes immediately
  • Can scale up or down as needed

Result: OpEx provides 5x faster tax benefits plus flexibility. This is why CFOs approve cloud migrations.

How SaaS/Cloud Companies Benefit from This

This creates a powerful business dynamic for cloud providers like AWS, Salesforce, and Snowflake:

The SaaS Sales Advantage

  • Easier to sell: "This is an operating expense, fully deductible this year" is a compelling pitch
  • Lower approval threshold: OpEx often doesn't require board approval; CapEx usually does
  • Faster sales cycles: No need for multi-year depreciation planning
  • Predictable revenue: Subscription model = recurring revenue (investors love this)

The Lock-In Effect

  • Switching costs: Once you build on AWS, migrating is expensive (data transfer, retraining, re-architecting)
  • Incremental expansion: Start with S3, then add Lambda, then RDS, then SageMaker... suddenly you're spending $1M/year
  • Consumption-based pricing: As your business grows, AWS revenue grows automatically

Why AWS is So Profitable

  • Economies of scale: AWS buys servers in bulk, depreciates them, then rents them to you at a markup
  • High margins: AWS operating margin is ~30% (vs. Amazon retail at ~5%)
  • Customer lifetime value: Average enterprise customer stays for 5+ years, spending increases over time

The Bottom Line for Data Scientists

Understanding OpEx vs. CapEx helps you:

  • Make better architecture decisions: "Should we buy a GPU server or use SageMaker?" ‚Äî Consider tax implications
  • Communicate with stakeholders: When proposing cloud solutions, mention the OpEx benefit to finance teams
  • Understand vendor incentives: Why does AWS push you toward more services? Because it increases their recurring revenue
  • Negotiate better: Know that cloud providers have high margins and room to discount for committed spend

Part 5: Amazon S3 ‚Äî The Complete Guide

Now let's dive deep into S3 (Simple Storage Service) ‚Äî AWS's foundational object storage service and the backbone of most cloud data workflows.

What is S3?

S3 (Simple Storage Service) is an object storage service that stores data as objects within buckets. Think of it as a massive, infinitely scalable hard drive accessible via HTTP/HTTPS.

Key characteristics:

  • Object storage: Files are stored as objects (not a file system)
  • Globally unique bucket names: Your bucket name must be unique across ALL of AWS
  • Regional storage: Data is stored in a specific AWS region (e.g., us-east-1, eu-west-1)
  • Virtually unlimited capacity: No storage limits (pay for what you use)
  • 99.999999999% durability: AWS replicates your data across multiple facilities

Core Concepts: Buckets and Objects

Buckets

  • What: Top-level containers for objects (like a folder, but not quite)
  • Naming: Must be globally unique, lowercase, 3-63 characters, DNS-compliant
  • Examples: my-data-lake-2026, ncf-student-websites
  • Region-specific: Created in a specific region, data doesn't leave unless you move it
  • Limit: 100 buckets per AWS account (soft limit, can request increase)

Objects

  • What: The actual files you store (images, videos, CSVs, HTML, etc.)
  • Key: The object's name/path (e.g., data/2026/sales.csv)
  • Value: The file content (up to 5 TB per object)
  • Metadata: Key-value pairs describing the object (content-type, custom tags)
  • Version ID: If versioning is enabled, each update creates a new version

S3 URL Structure

https://BUCKET-NAME.s3.REGION.amazonaws.com/OBJECT-KEY

Example:
https://my-website-bucket.s3.us-east-1.amazonaws.com/index.html
https://my-data-lake.s3.us-west-2.amazonaws.com/data/2026/sales.csv

What S3 CAN Do

‚úÖ Storage Capabilities

  • Store any file type: Images, videos, documents, code, databases, ML models
  • Massive scale: Single objects up to 5 TB, unlimited total storage
  • Static website hosting: Serve HTML/CSS/JS directly from S3 (what we'll do in the lab!)
  • Data lakes: Store petabytes of raw data for analytics
  • Backup and archival: Long-term storage with Glacier storage classes

‚úÖ Access and Delivery

  • HTTPS access: All S3 endpoints support HTTPS (encrypted in transit)
  • Public or private: Control access with bucket policies and IAM
  • Pre-signed URLs: Temporary access to private objects (expires after X seconds)
  • CDN integration: Use CloudFront to cache content globally
  • Cross-region replication: Automatically copy objects to other regions

‚úÖ Data Management

  • Versioning: Keep multiple versions of objects (undo deletes/overwrites)
  • Lifecycle policies: Auto-delete or move to cheaper storage after X days
  • Encryption: Server-side encryption (SSE-S3, SSE-KMS) or client-side
  • Event notifications: Trigger Lambda functions when objects are created/deleted
  • Object tagging: Organize and track costs by project/team

‚úÖ Integration with AWS Services

  • Athena: Query data in S3 using SQL (no loading required)
  • Lambda: Process objects automatically (resize images, transform data)
  • SageMaker: Read training data from S3, save models to S3
  • EMR/Spark: Distributed processing of S3 data
  • Glue: ETL jobs reading from and writing to S3

What S3 CANNOT Do

‚ùå Not a File System

  • No directories: S3 has a flat structure (keys can contain / but they're just part of the name)
  • No partial updates: Can't edit a file in place ‚Äî must upload the entire object again
  • No file locking: Multiple writers can overwrite each other (last write wins)
  • No POSIX operations: Can't cd, mv, or chmod like a Linux filesystem

‚ùå Not a Database

  • No SQL queries: Can't query object contents directly (use Athena for that)
  • No indexing: Can't search by object content, only by key name
  • No transactions: Can't atomically update multiple objects
  • No joins: Can't relate objects to each other

‚ùå Not for Dynamic Web Applications

  • Static content only: Can serve HTML/CSS/JS, but can't run server-side code (PHP, Python, Node.js)
  • No server-side processing: Can't connect to databases, can't handle form submissions
  • No custom domains with HTTPS (directly): Need CloudFront for https://yourdomain.com
  • No WebSockets: Can't maintain persistent connections

‚ùå Performance Limitations

  • Eventual consistency (for some operations): Overwrites/deletes may take seconds to propagate
  • Latency: First-byte latency is ~100-200ms (use CloudFront for faster delivery)
  • Request rate limits: 3,500 PUT/COPY/POST/DELETE and 5,500 GET/HEAD requests per second per prefix

S3's RESTful API: How You Actually Interact with S3

S3 is accessed via a RESTful HTTP API. Every operation (upload, download, delete) is an HTTP request. Understanding this is crucial for working with S3 programmatically.

Common S3 API Operations

1. Upload an object (PUT)

PUT /my-file.txt HTTP/1.1
Host: my-bucket.s3.us-east-1.amazonaws.com
Content-Type: text/plain
Authorization: AWS4-HMAC-SHA256 Credential=...

[file content here]

2. Download an object (GET)

GET /my-file.txt HTTP/1.1
Host: my-bucket.s3.us-east-1.amazonaws.com
Authorization: AWS4-HMAC-SHA256 Credential=...

3. List objects in a bucket (GET)

GET /?list-type=2&prefix=data/ HTTP/1.1
Host: my-bucket.s3.us-east-1.amazonaws.com
Authorization: AWS4-HMAC-SHA256 Credential=...

4. Delete an object (DELETE)

DELETE /my-file.txt HTTP/1.1
Host: my-bucket.s3.us-east-1.amazonaws.com
Authorization: AWS4-HMAC-SHA256 Credential=...

5. Get object metadata (HEAD)

HEAD /my-file.txt HTTP/1.1
Host: my-bucket.s3.us-east-1.amazonaws.com
Authorization: AWS4-HMAC-SHA256 Credential=...

How You Actually Use S3 (You Don't Write Raw HTTP)

In practice, you use tools that wrap the REST API:

1. AWS CLI (Command Line Interface)

# Upload a file
aws s3 cp myfile.txt s3://my-bucket/

# Download a file
aws s3 cp s3://my-bucket/myfile.txt ./

# List objects
aws s3 ls s3://my-bucket/

# Sync a directory (like rsync)
aws s3 sync ./my-website s3://my-bucket/

2. AWS SDKs (Python, JavaScript, Java, etc.)

# Python (boto3)
import boto3
s3 = boto3.client('s3')

# Upload
s3.upload_file('myfile.txt', 'my-bucket', 'myfile.txt')

# Download
s3.download_file('my-bucket', 'myfile.txt', 'downloaded.txt')

# List
response = s3.list_objects_v2(Bucket='my-bucket')
for obj in response['Contents']:
    print(obj['Key'])

3. AWS Console (Web UI)

  • Point-and-click interface for managing buckets and objects
  • Good for learning and one-off tasks
  • Not recommended for production workflows (use CLI/SDK instead)

S3 Storage Classes: Optimizing for Cost

S3 offers different storage classes with different pricing and performance characteristics:

Storage Class Use Case Cost (per GB/month) Retrieval Time
S3 Standard Frequently accessed data $0.023 Instant
S3 Intelligent-Tiering Unknown/changing access patterns $0.023 + monitoring fee Instant
S3 Standard-IA Infrequently accessed (backups) $0.0125 Instant
S3 Glacier Instant Retrieval Archive with instant access $0.004 Instant
S3 Glacier Flexible Retrieval Long-term archive $0.0036 Minutes to hours
S3 Glacier Deep Archive Compliance/regulatory (7-10 years) $0.00099 12-48 hours

Prices are approximate for us-east-1 region as of 2026. Check AWS S3 Pricing for current rates.

Cost Optimization Tip

Use lifecycle policies to automatically transition objects to cheaper storage classes:

  • Keep data in S3 Standard for 30 days (frequent access)
  • Move to S3 Standard-IA after 30 days (occasional access)
  • Move to Glacier after 90 days (archive)
  • Delete after 365 days (if no longer needed)

Example savings: 1 TB stored for 1 year

  • S3 Standard all year: $276
  • With lifecycle policy: ~$100 (64% savings!)

S3 Security and Access Control

S3 has multiple layers of security. Understanding these is critical for both protecting your data and making it accessible when needed.

1. Bucket Policies (Resource-Based)

JSON policies attached to buckets that control who can access what.

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": "*",
    "Action": "s3:GetObject",
    "Resource": "arn:aws:s3:::my-website-bucket/*"
  }]
}

Translation: Allow anyone (Principal: "*") to read (GetObject) all objects in this bucket (for public websites).

2. IAM Policies (User-Based)

Policies attached to IAM users/roles that grant permissions.

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": ["s3:GetObject", "s3:PutObject"],
    "Resource": "arn:aws:s3:::my-data-bucket/*"
  }]
}

Translation: This user can read and write objects in my-data-bucket.

3. Block Public Access Settings

  • Default: All new buckets block public access (AWS learned from data breaches)
  • For static websites: You must explicitly disable this setting
  • Best practice: Only make public what needs to be public

4. Pre-Signed URLs (Temporary Access)

Generate time-limited URLs for private objects:

# Generate a URL that expires in 1 hour
aws s3 presign s3://my-bucket/private-file.pdf --expires-in 3600

# Result:
https://my-bucket.s3.amazonaws.com/private-file.pdf?
  X-Amz-Algorithm=AWS4-HMAC-SHA256&
  X-Amz-Credential=...&
  X-Amz-Expires=3600&
  X-Amz-Signature=...

Use case: Share a private file with someone without making the whole bucket public.

5. Encryption

  • In transit: Always use HTTPS (S3 supports both HTTP and HTTPS, but use HTTPS)
  • At rest: Server-side encryption (SSE-S3, SSE-KMS, SSE-C) or client-side encryption
  • Default: As of 2023, all new objects are encrypted by default (SSE-S3)

S3 Pricing: What You Actually Pay For

Three Main Cost Components:

1. Storage ($0.023/GB/month for S3 Standard)

  • Charged for the amount of data stored
  • Calculated daily, billed monthly
  • Example: 100 GB for 1 month = $2.30

2. Requests ($0.0004 per 1,000 PUT requests, $0.0004 per 10,000 GET requests)

  • Charged per API call (upload, download, list, delete)
  • GET/HEAD requests are cheaper than PUT/POST/DELETE
  • Example: 1 million GET requests = $0.40

3. Data Transfer ($0.09/GB out to internet, free within AWS region)

  • Free: Data transfer INTO S3
  • Free: Data transfer between S3 and EC2 in the same region
  • Paid: Data transfer OUT to the internet ($0.09/GB for first 10 TB)
  • Example: Serve 1 TB of images to users = $90

Real-World Cost Example: Static Website

Scenario: Personal portfolio website

  • Website size: 50 MB (HTML, CSS, JS, images)
  • Traffic: 1,000 visitors/month, 10 pages each = 10,000 requests
  • Data transfer: 50 MB √ó 1,000 visitors = 50 GB out

Monthly cost:

  • Storage: 0.05 GB √ó $0.023 = $0.001
  • Requests: 10,000 GET √ó $0.0004/10,000 = $0.0004
  • Data transfer: 50 GB √ó $0.09 = $4.50
  • Total: ~$4.50/month

With CloudFront CDN: ~$0.50/month (CloudFront has cheaper data transfer rates)
