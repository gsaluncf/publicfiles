Strong Entities
  • Have their own unique identifying key
  • Can exist independently
  • Represent real-world "things" or objects
  • Examples: Customer, Product, Employee
  • Business-level tangible items
  • Single column primary key (usually)
üîó
Weak Entities
  • Depend on another entity for existence
  • Have composite keys linking other tables
  • Exist primarily for database structure
  • Examples: OrderItem, FilmCategory, Enrollment
  • Hidden entities discovered during modeling
  • Used for JOIN operations
üè∑Ô∏è Naming Conventions for Junction Tables

Good naming conventions help identify table purposes and relationships

Pattern: Entity1_Entity2
film_category
student_course
author_book
employee_project

Purpose: Immediately identifies the two entities being joined

Pattern: BusinessProcess
enrollment (student + course)
order_item (order + product)
booking (passenger + flight)
assignment (employee + task)

Purpose: Describes the business relationship/process

Pattern: EntityAction
student_enrollment
customer_purchase
employee_assignment
user_login

Purpose: Emphasizes the action or event being tracked

ü§î Decision Process: Strong vs Weak Entity
1 Does this entity have its own unique, meaningful identifier?
If YES ‚Üí Likely a Strong Entity
If NO ‚Üí Continue to step 2
2 Can this entity exist without referencing other entities?
If YES ‚Üí Likely a Strong Entity
If NO ‚Üí Continue to step 3
3 Does this entity primarily exist to connect other entities?
If YES ‚Üí Likely a Weak Entity (Junction Table)
If NO ‚Üí Continue to step 4
4 Does this entity represent a real-world "thing" vs. a relationship?
Real-world thing ‚Üí Strong Entity
Relationship/Process ‚Üí Weak Entity
üéØ Practice: Identify Entity Types
Scenario 1: A library system needs to track which books are written by which authors. Some books have multiple authors, and some authors have written multiple books.
Scenario 2: An e-commerce system needs to track individual products in customer orders, including quantity and price at time of purchase.
Scenario 3: A movie database needs to track which actors appeared in which films, including their role/character name.
